Spring程序的分层

前端–>Controller–>Service–>Dao–>DataBases–>Dao–>Service–>Controller–>前端

3层架构

Controller：控制层；主要负责和前端的交互，即接收前端的数据，响应前端的数据。

Service：业务逻辑层；处理具体的业务逻辑，比如我收到一个学生的体检信息（身高体重），我可以通过Service写个函数判断BMI（这人偏胖/瘦/正常。。。）

Dao：（Data Access Object ）持久层/数据层；负责数据的访问操作（增删改查）

图片中的三层架构还不是最终形态，还没有进行ioc（控制反转）和id（依赖注入），关于这两方面很快会讲。

简单分层后的问题

可以看到在3层中，每次要调用其他层的方法都需要先进行导入，虽然使用了多态的写法private UserService userService = new UserServiceImpl();，但是如果我要使用UserServiceImpl2的实现类呢，或者UserServiceImpl3，那我除了要写这些新的UserService接口的实现类，还需要去修改Controller中的代码，这维护起来显然有些麻烦。Spring解决了这个问题，引入了两个特性IoC和DI。

IoC 控制反转 Inversion of Control

一般情况下，我们要实例化一个对象，肯定使用Object ob = new Object();这种写法在类内部主动创建依赖对象，但是有了IoC之后，我们把创建依赖对象的权力交给了IoC容器。

DI 依赖注入 Dependency Injection

IoC容器为应用程序提供运行时，所依赖的资源，称之为依赖注入。

Bean对象

顾名思义，就是在IoC容器中，像一个豆荚里的豆子，IoC容器是豆荚。

图中用@Component注解了UserServiceImpl和UserDaoImpl，这样就相当于把这个类交给IoC容器，由IoC容器来控制。在其他类中需要实例化接口（比如private UserService userService;）时，在创建实例化对象的语句上加上@Autowired注解，Spring会自动去找接口对应的实现类。

以private UserService userService;为例，在UserController调用private UserService userService;并加上@Autowired注解，运行时，Spring会自动寻找实现了UserService接口的实现类，找到了有@Component的UserServiceImpl，那初始化就使用这个实现类进行初始化。

用@Component和@Autowired的好处就是，比如我需要修改实现类，我现在要用NewUserServiceImpl而不是UserServiceImpl，我只要去掉UserServiceImpl的@Component注解，在NewUserServiceImpl里加上注解就行，不用修改UserController，因为类似private UserService userService = new UserServiceImpl();是由Spring去控制的，不用你操心

这里需要对@Component注解进行一个说明，在Spring中，component注解有3个衍生注解，对应3层架构，具体如下

注解 说明 位置
@Component 声明bean的基础注解 不属于以下三类时，用此注解
@Controller @Component的衍生注解 标注在控制层类上
@Service @Component的衍生注解 标注在业务层类上
@Repository @Component的衍生注解 标注在数据访问层类上（由于与mybatis整合，用的比较少）
